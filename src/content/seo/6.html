Core Web Vitals 優化指南：LCP、CLS、INP 全面提升

掌握關鍵效能瓶頸與實作細節，改善載入速度、互動延遲與版面位移

摘要：掌握關鍵效能瓶頸與實作細節，改善載入速度、互動延遲與版面位移，提升整體體驗與排名機會。

導言

想讓網頁「打開快、點了有反應、畫面不亂跳」？Core Web Vitals（CWV）就是 Google 用來衡量這三件事的指標組合：LCP（最大內容出現時間）、CLS（版面位移）、INP（互動到下一次繪製）。通過這三關，使用者更舒服，搜尋表現也較有機會穩定。(Google for Developers)

1. 定義與為何重要

1.1 三個指標，一句話抓重點

  • LCP：主要內容「看得到」要花多久。(web.dev - LCP)
  • CLS：元素亂移位的程度（無單位分數）。(web.dev - CLS)
  • INP：使用者點擊、輸入後，畫面多久有回饋。(web.dev - INP)

1.2 與搜尋的關係

Google 建議站長把 CWV 維持在良好範圍；Search Console 會用真實使用者資料（CrUX）出報表，幫你找出問題頁。(Core Web Vitals 報表)

1.3 一個關鍵變動

INP 已在 2024/3 取代 FID 成為 Core Web Vitals 的互動性指標；Search Console 與相關工具已陸續更新。(Google: Introducing INP)

本段重點：三指標對應三種體驗；通過門檻有助於整體表現，且 INP 已成為互動性衡量標準。

2. 原理與流程（載入 → 互動 → 穩定）

2.1 標準值（75 百分位，分行動/桌機）

  • LCP ≤ 2.5s 為佳；4.0s 以上為差。(Optimize LCP)
  • INP ≤ 200ms 為佳；>500ms 為差。(INP 指標說明)
  • CLS ≤ 0.1 為佳；>0.25 為差。(Optimize CLS)

2.2 為何工具數字會不一樣？

PageSpeed Insights（PSI）會同時呈現「實際資料（CrUX）」與「實驗室量測（Lighthouse）」。情境不同、測法不同，數字也會不同——所以要同時看兩者，並以 75 百分位場景為優先判斷依據。(About PageSpeed Insights)

本段重點：以 75 百分位為門檻；理解場景 vs. 實驗室差異，才能找到真問題。

3. 三大面向：量測 / 前端實作 / 伺服器與資源

3.1 量測與診斷（先找症結）

  • Search Console：看整站/分群問題與受影響頁面。
  • PageSpeed Insights：單頁場景 + 實驗室分析。
  • DevTools / Lighthouse：拆解阻塞資源、字型、圖片與 JS 長任務。

先量後修，優先處理影響最多人的瓶頸。

3.2 前端實作（使用者看得到的部分）

  • LCP：確認首屏最大元素（通常是主圖或大標）並優化其載入：預載、壓圖、避免巨型背景圖。
  • CLS：為所有媒體/廣告保留尺寸；使用容器比例佔位；處理字型 FOUT/FOIT。
  • INP：避免主執行緒長任務，將非必要 JS 延後，互動後立即給回饋（loading 或骨架）。

把「首屏最大元素」「會亂動的東西」「卡主執行緒的程式」先解決。

3.3 伺服器與資源（底層速度）

  • 快取與 CDN、壓縮回應以優化 TTFB。
  • 採用 HTTP/2 或 HTTP/3，並按需載入資源（critical CSS、code-splitting）。
  • 圖片管線：WebP/AVIF、多尺寸輸出、自動轉換及 lazy-load 非首屏圖。

上述手法目的：讓 LCP 更快、INP 更順、CLS 不亂跳。

4. 入門步驟（照流程修：LCP → CLS → INP）

4.1 先救 LCP（最大內容出現時間）

清單

  • 找出首屏最大的元素（通常主圖或主標）並優先優化。
  • 圖片：使用 WebP/AVIF、輸出對應尺寸、移除阻塞樣式。
  • 樣式與字型：將關鍵 CSS 內嵌；使用字型顯示策略避免首屏空白。
  • JS：把非必要腳本設為 defer 或延後載入。

參考：web.dev LCP 與優化指南。

4.2 再穩 CLS（版面位移）

清單

  • 所有圖片/影片/iframe/廣告預先保留寬高或使用容器比例。
  • 避免在已渲染內容上方插入新元素（例如動態訂閱條、跑馬燈）。
  • 使用 font-display 與字重替換策略減少版面跳動。

參考：web.dev CLS 與常見肇因。

4.3 最後攻 INP（互動反應）

清單

  • 拆分長任務（>50ms），把大工作拆成小塊。
  • 事件處理器保持精簡，將重計算交給 Web Worker。
  • 第三方 SDK/廣告/分析延後載入或按需載入。

參考：web.dev INP 與最佳做法。

4.4 持續監測與回歸

  • 每週檢查 Search Console 的 CWV 報表；針對「需要改進/差」群組建立工單。
  • 每次上線前跑 PSI，並在 CI 中納入 Lighthouse 檢查以避免回歸。

5. 常見誤區 & 實用準則

5.1 常見誤區

  • 只追 Lighthouse 分數，真實使用者資料（CrUX）沒改善。
  • 圖片只壓縮但沒裁切，首屏仍載入過大檔案。
  • 未為嵌入物保留尺寸，導致 CLS 增高。
  • 長任務卡主主執行緒，互動延遲高卻只怪框架。

5.2 實用準則

  • 先大後小：優先處理影響最多人的頁型與版位。
  • 首屏為王：LCP/CLS 常在第一眼發生。
  • 互動要回話：即使結果需等待，也要先給使用者狀態或骨架。
  • 每季健檢：建立固定檢視與回歸測試節奏。

7. 案例流程（一步步演練）

情境：圖文型內容站的文章頁，行動端 LCP 偏慢、CLS 偶發、INP 屬中間區。

里程碑

  1. 盤點：Search Console 找出「差/需改進」群組 → 選最高流量的 10 篇優先修復。
  2. LCP 衝刺：主圖改 WebP，按裝置輸出對應尺寸；關鍵 CSS 內嵌；延後第三方按鈕載入。
  3. CLS 歸零：所有媒體、廣告位使用固定尺寸容器或比例佔位；避免在頁首動態插入元素。
  4. INP 瘦身：拆分重腳本、互動先回饋再執行重流程、使用 requestIdleCallback 處理非緊急任務。
  5. 驗收：PSI 對照前後，並觀察一週後的 CrUX 變化，持續迭代。

結語

CWV 不是一次交付的專案，而是長期維運的節拍：先量測 → 抓大石頭 → 小步快跑 → 固定回看。當首屏出得快、畫面不晃、互動有回應，使用者會更願意停留與深度互動，這些行為也會在搜尋系統中慢慢累積成優勢。今天就挑一個頁型，照第 4 章的清單做一次，你會很快看到體感變順、指標跟著上來。
