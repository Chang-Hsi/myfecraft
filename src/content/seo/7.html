結構化資料與豐富結果：Article、FAQ、Breadcrumb 標記

用結構化資料幫搜尋引擎讀懂頁面意圖，增加出現精選摘要與豐富結果的機會

摘要：用結構化資料幫搜尋引擎讀懂頁面意圖，增加出現精選摘要與豐富結果的機會。

導言

同一篇內容，用對「結構化資料（Structured Data）」就像在頁面上貼上小抬頭，告訴搜尋系統：這是文章、這裡是作者、這段是問題與回答、這條是導覽路徑。理解越清楚，越有機會在搜尋結果呈現更豐富的樣式（Rich Results），例如更完整的標題、縮圖、路徑等。本文聚焦三種最常用的標記：Article、FAQPage、BreadcrumbList，並給你可直接套用的 JSON-LD 範例與常見地雷。(Google for Developers)

1. 先釐清：結構化資料與豐富結果的關係

  • 結構化資料是語意標記：用 JSON-LD、Microdata 或 RDFa 描述頁面要素（建議用 JSON-LD）。(General Structured Data Guidelines)
  • 豐富結果是呈現方式：是否顯示由 Google 決定；就算標得正確，也不保證一定出現。(Intro to Structured Data)
  • 內容要能對上畫面：標記不可捏造或標示看不到的資訊，違規會被忽略或處罰。

延伸閱讀：Structured data 基本原理與支援項目總覽。

2. Article：讓系統正確理解你的文章

什麼時候用

新聞、部落格、專題報導、教學文等適合用 Article（NewsArticle、BlogPosting 亦可）。系統能更準確抓到標題、主圖、時間與作者資訊。(Article 指南)

必備/常用欄位建議

  • headline、image（至少 1200px 寬的代表圖）、datePublished、dateModified、author、publisher、mainEntityOfPage

JSON-LD 範例（可直接改值）

{
  "@context": "https://schema.org",
  "@type": "Article",
  "mainEntityOfPage": { "@type": "WebPage", "@id": "https://example.com/blog/structured-data" },
  "headline": "結構化資料與豐富結果實作指南",
  "image": ["https://example.com/images/cover-1200.jpg"],
  "datePublished": "2025-09-10",
  "dateModified": "2025-09-12",
  "author": { "@type": "Person", "name": "王小明" },
  "publisher": {
    "@type": "Organization",
    "name": "Example Inc.",
    "logo": { "@type": "ImageObject", "url": "https://example.com/logo-120x120.png" }
  },
  "description": "用 JSON-LD 幫搜尋引擎理解文章，提升顯示品質。"
}

實作提示：發佈與更新時間請確實；主圖清晰且比例合理。Google News/Discover 對時間與圖片亦有注意事項。

3. FAQPage：Q&A 內容的標記（顯示規則已收緊）

什麼時候用

頁面中有多組「問題＋回答」且由同一提供者撰寫（非使用者留言），適合使用 FAQPage。(FAQPage 指南)

重要注意

自 2023 年後，FAQ 豐富結果顯示範圍收緊，系統會更傾向於政府或健康等可信站點；其他網站即使標記正確也未必顯示。

JSON-LD 範例

{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "什麼是結構化資料？",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "用標準化格式描述頁面元素，幫搜尋系統快速理解內容。"
      }
    },
    {
      "@type": "Question",
      "name": "FAQ 標記是不是一定會有加強顯示？",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "不一定。Google 目前多半只在政府與健康領域的權威網站顯示 FAQ 豐富結果。"
      }
    }
  ]
}

實作提示：問題與回答務必與實際可見內容一致，勿塞關鍵字或行銷語；一頁放置多組 Q&A 才適合用 FAQPage。

4. BreadcrumbList：清楚的階層路徑

為何要用

導覽路徑標記表明頁面在網站中的層級位置，有助於系統理解結構，並在某些情境中作為顯示依據。(Breadcrumb 指南)

JSON-LD 範例

{
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    { "@type": "ListItem", "position": 1,
      "name": "首頁", "item": "https://example.com/" },
    { "@type": "ListItem", "position": 2,
      "name": "知識庫", "item": "https://example.com/knowledge/" },
    { "@type": "ListItem", "position": 3,
      "name": "結構化資料實作", "item": "https://example.com/knowledge/structured-data" }
  ]
}

實作提示：至少兩個 ListItem，每個項目需有 name 與對應 URL；避免路徑與實際導覽不一致。

5. 放到頁面前，請先做這三件事

  1. 技術規範：採用 JSON-LD；不要用 robots.txt/noindex 把含標記的頁面擋掉。
  2. 品質規範：標記要和實際畫面一致，不過度包裝、不誤導。
  3. 測試：先用 Rich Results Test 或 Search Console 的 URL 檢查驗證語法與可見性。

6. 常見地雷（避免踩）

  • 看不到卻標上去：例如把隱藏區塊或彈窗內容拿來標；易被判定品質不佳。
  • FAQ 亂用：把行銷文硬切成 Q&A；現在多數站點也不會得到 FAQ 富顯示。
  • 路徑不一致：Breadcrumb 與實際導覽不符或缺最後一階 URL。
  • 圖片太小：Article 的代表圖過小，可能不符合顯示條件（建議至少 1200px 寬）。

7. 實作步驟（一步步落地）

  1. 盤點頁型：列出文章頁、FAQ 集中頁、分類頁，決定應用哪一種標記。
  2. 設計欄位：把標題、主圖、作者、時間、路徑對應到 CMS 欄位。
  3. 輸出 JSON-LD：在 <head> 或頁尾輸出；以伺服器端或前端渲染皆可。
  4. 驗證與上線：批次以 Rich Results Test 驗證；上線後用 Search Console 看偵測情況與告警。

8. 範例組合：文章頁 + FAQ 區塊 + 導覽路徑

在一篇教學文中，你可以同時放 Article（描述整篇）、頁內的「常見問題」區塊用 FAQPage（若真的以 Q&A 呈現）、頂部或麵包屑用 BreadcrumbList。三者互不衝突，但請確保 內容都在頁面上看得到。(相關說明)

結語

結構化資料不是「加了就會發光」，而是把內容用標準格式說清楚。先把 Article、FAQPage、BreadcrumbList 做正確，並確保頁面真的提供這些資訊；後續再按頁型擴充（例如產品、活動、評分…）。把基本功打好，搜尋系統更容易理解你，讀者也更快找到重點。
